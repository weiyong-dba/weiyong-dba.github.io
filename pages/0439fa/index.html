MySQL基于MGR+PROXYSQL的HA方案

# 前言

MySQL Group Replication（后面称为MGR）是对数据最终一致性（注意，是数据最终一致性，实时？做不到，还是有一定延迟）的保证还是挺靠谱的。MGR有两种模式，单主和多主，针对当前多主的限制，多主的实用性应该还不大。 实际上，通过MGR单主模式已经可以消除传统的主从架构故障切换后数据可能不一致的隐患，除此之外还有一个问题，就是HA，如何做到MGR主节点故障自动切换。

MGR单主模式下，当主节点故障，MGR内部将会发起一轮选举，选出新的主，这是由MGR内部决定并执行的。但是应用的连接遇到主节点挂掉的情况，是不会自动发生切换的。也就是说，MGR内部没有提供一种机制，来实现主节点故障切换对应用无感知。意思就是说，GMR并不能帮你处理客户端的故障切换，这事得由我们应用自己来！又或者是依靠中间件、proxy这类软件！ 在实际应用中，我们希望，主节点挂掉，应用无需重启，自动能够将连接重置到新的主上，继续提供服务。基于这方面考虑，我们结合mysql中间件ProySQL来解决上面提到的问题。

# 目标

  • MGR单主模式下，依赖ProxySQL这款中间件实现主节点故障切换，对应用无感知。
  • 新的主节点需要应用完成所有binlog日志后才能提供服务，保证数据不会冲突。
  • 由于MGR需要3个节点，但实际应用中出于成本考虑，更多的是采用一主一从，因此，我这里提供一种解决方案就是把其中一个节点的所有表都更改为BLACKHOLE引擎，这样一来，这个节点就不会占用磁盘很多空间，保证了集群数据完整可用。

# 实现思路

实现思路图如下:

应用通过连接ProxSQL中间件，间接访问后端MGR主节点。ProxySQL内部有配置表，可以维护MGR节点的访问信息，并开启调度器调度检查脚本（Shell脚本），定期巡检后端MGR节点状态，若发现MGR主节点挂掉，ProxySQL调度脚本检测到这个错误，从而确定新的主节点，将原先持有的旧的连接丢弃，产生新节点的连接（ProxySQL内部会维护和后端MGR各个节点的连接，多源连接池的概念）。

上述的整个过程中，应用无需任何变动。应用从意识发生了故障，到连接重新指向新的主，正常提供服务，秒级别的间隔。

# 具体实施

下面介绍如何配置ProxySQL和MGR共同作用来达成我们的目标，MGR的部署请参考官网文档。

部署环境

  • CentOS7
  • MySQL 5.7.18
  • ProxySQL 1.3.7

配置

  • active : 1: 表示使脚本调度生效
  • interval_ms : 每隔多长时间执行一次脚本 (eg: 3000(ms) = 3s 表示每隔3s脚本被调用一次)

  • arg4 -> log file, default: './checker.log'

# 测试

写一段Java程序，通过JDBC连接，连接ProxySQL（注意，客户端应该连接的是ProxySQL），执行select @@hostname查看当前连着的是后端MGR哪个节点，然后手动模拟主机挂掉的情况，观察现象。代码如下:

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.ResultSet;
import java.sql.SQLException;
import java.sql.Statement;
import java.sql.PreparedStatement; 
 
public class TestMgrHAWithProxysql {
    private static final String JDBC_URL = "jdbc:mysql://172.x.x.1:6033/test";
    private static final String USER = "proxysql";
    private static final String PASSWORD = "proxysql";
    public static void main(String[] args) {
        tryAgain();
    }
 
    private static void tryAgain() {
        Connection conn = null;
        int i = 0;
        try {
           conn = DriverManager.getConnection(JDBC_URL, USER, PASSWORD);
            conn.setAutoCommit(false);
            String sql = "select @@hostname as hostname";
            String sql3 = "set autocommit=1";
            String sql2 = "insert into test.a(c,b) values (?,?)";
            Statement stmt = conn.createStatement();
            stmt.executeQuery(sql3);
            PreparedStatement PStatement = conn.prepareStatement(sql2);   
            int j = 0;
            while(true) {
                j=j+1;
                System.out.println("j: " + j);
                PStatement.setInt(1, j);
                ResultSet rs = stmt.executeQuery(sql);
                while (rs.next()) {
                        String hostname = rs.getString("hostname");
                        System.out.println("hostname: " + hostname);
                PStatement.setString(2, hostname);
                i = PStatement.executeUpdate(); 
                }
                System.out.println("insert : " + i);
                if(rs.next()) {
                    System.out.println("port : " + rs.getString(1));
                }
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        } catch(SQLException e) {
            e.printStackTrace();
            tryAgain();
        } finally {
            if(conn != null) {
                try {
                    conn.close();
                } catch (SQLException e) {
                    e.printStackTrace();
                }
            }
        }
    }
}

# 引入ProxySQL的影响

本方案是通过引入中间件，来解决MGR单主模式下主发生故障切换，对应用无感知。有以下影响:

  • 引入中间件proxysql，增加机器资源的同时增加了运维难度和复杂度
  • proxysql存在单点，如果要做到高可用，需要KeepAlived支持，但这也很容易做到。
  • MGR性能损失20%
