MySQL高可用架构之PXC

# MySQL高可用架构之PXC

# PXC主要特性

  • 同步复制，没有数据延迟。
  • 多主复制，所有节点都可以读写。
  • 行级别并行复制。
  • 新增节点自动完成同步。
  • 所有节点数据强一致性。
  • Configuration script for ProxySQL
  • Automatic configuration of SSL encryption
  • 兼容MySQL Server Community Edition, Percona Server, and MariaDB。

# PXC主要限制

  • 仅支持innodb引擎，因此操作mysql.*系统表需要用DDL而不能用DML。
  • LOCK TABLES和UNLOCK TABLES在multi-master模式下不支持。
  • Lock functions, such as GET_LOCK(), RELEASE_LOCK(), and so on。
  • Query log不能写入表，只能写入文件。
  • Maximum allowed transaction size is defined by the wsrep_max_ws_rows and wsrep_max_ws_size variables. LOAD DATA INFILE processing will commit every 10 000 rows. So large transactions due to LOAD DATA will be split to series of small transactions.
  • 乐观锁
  • XA transactions are not supported due to possible rollback on commit.
  • 集群性能由最差节点决定。
  • 建议3个节点，第三个节点可以是仲裁节点。
  • InnoDB fake changes feature is not supported.
  • enforce_storage_engine=InnoDB is not compatible with wsrep_replicate_myisam=OFF (default).
  • 不支持传输表空间
  • 必须有主键

# 安装

在3个node上执行安装。

yum install https://www.percona.com/downloads/percona-release/redhat/1.0-13/percona-release-1.0-13.noarch.rpm -y
yum install Percona-XtraDB-Cluster-57 -y

# 启动

  1. 启动第一个节点 - systemctl start mysql@bootstrap.service
  2. 从error日志里面找到临时密码，登录mysql修改密码 - set password=password('123456');
  3. 查看状态 - show status like 'wsrep%';
  4. 创建复制用户
mysql@pxc1> CREATE USER 'sstuser'@'localhost' IDENTIFIED BY 'passw0rd';
mysql@pxc1> GRANT RELOAD, LOCK TABLES, PROCESS, REPLICATION CLIENT ON *.* TO
  'sstuser'@'localhost';
mysql@pxc1> FLUSH PRIVILEGES;

  1. 启动第二个节点 - systemctl start mysqld.service
  2. 等待第二个节点的 - wsrep_local_state_comment变成Synced
  3. 启动第三个节点 - systemctl start mysqld.service

# 基准测试

  1. 准备环境 - create database proxytest default character set utf8;

  2. 执行测试 - sysbench --test=/usr/share/doc/sysbench/tests/db/oltp.lua --mysql-host=node1 --mysql-port=3306 --mysql-user=proxysql --mysql-password=proxysql --mysql-db=proxytest --oltp-tables-count=10 --oltp-table-size=1000000 --report-interval=20 --oltp-dist-type=uniform --rand-init=on --max-requests=0 --oltp-read-only=off --max-time=120 --num-threads=16 prepare

# 仲裁节点

PXC双节点由于脑裂的发生和quorum机制，通常单纯双节点的percona集群无法保证环境的可靠性。因此引入了仲裁技术-->arbitrator来防止双节点的集群环境中由于通信故障而导致节点无法正常同步的情况发生。当节点无法接受到其他节点的通信时，如果可以同arbitrator连接，可以通过arbitrator接受和发送同步数据。arbitrator不应用接收的同步数据，但需要查看所有的同步活动，因此arbitrator所在的网络速度慢的话会导致整个集群性能下降。
